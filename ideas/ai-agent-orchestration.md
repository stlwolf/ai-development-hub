# AIエージェント・オーケストレーションに関するアイデアメモ

## 1. 自作オーケストレーションの可能性

ある程度今の技術で、自分でサービス化してオーケストレーションに近いものを作ることは可能か？

## 2. シンプルな共有コンテキストによるアプローチ

単純にAPI経由でのやり取りでも、時間的コストがかかるとはいえ可能ではないか。

どこかにプロンプトや情報を集約するものを作って、常にやり取りをそこに集約することで、相互参照するだけでもオーケストレーションに近いことは可能なのではないかと思った。

## 3. 文脈と履歴を理解するガイドAgent

### 問題意識

特定の役割を持つAgent（例：DB用のAgent、社内ナレッジを検索するためのDBエージェント）のように、一つの集約されたドメインやAI情報のインデックスを貯めていくのは、おそらく鉄板的なデータのやり方だと思う（そもそもトレーニングデータがそういうもの）。

しかし、それが増えれば増えるほどコンテキストの理解が難しくなる。一時的なメモリ（コンテキストの最大量）は増えているとはいえ、増えれば増えるほど何をやっているかわからなくなるし、検索するためのコストもかかるし、正しい情報にたどり着けない。これはAI側でも同様に起きると思う。

### アイデア

そういったドメインを検索するための「ガイドライン」に特化したAIモデルが、オーケストレーションの中枢にいる必要があるのではないか。

静的な情報というよりは、常に増え続ける共有データの中で、どういった文脈でそれが追跡されてきたのか、蓄積されてきたのかを前提として持ちつつ、必要な情報を渡すもの、都度渡すもの、というイメージ。

今見ているのは、おそらく社内ナレッジのDB、いわゆるRAGで、どれだけその文脈に沿ったものを返すかというところだと思う。しかし、その「文脈」というのが、歴史的なところからもアプローチして教えてくれることが重要だと思う。

つまり、1ヶ月前に自分が書いたコードは他人が書いたコードと同じで、蓄積されたデータも結局意味がわからなくなってしまう。その辺の履歴的なところも追わないといけないと思うが、その辺をガイドラインしてくれる役割、というイメージ。

## 4. AI開発のやりとり自体を資産化する

ふと思ったのが、結局最初からAIで開発したものはプロンプトに全て集約されている。

ある程度そのプロンプトをブラッシュアップして保存するということはあり得るとしても、**そのやりとり自体が歴史的経緯を持つデータになる**ので、それをそのままの形で保存しつつ、増えてきた時に役立つ文脈データとして持つことが重要ではないか。

それを「AIエージェントツールを使って開発してきたもの」として、常に裏側に送るような仕組みを自分たちの開発フローで作ってしまい、蓄積していく。そして、そういったデータで学習したモデルで、先述の「モデレーターAI」のモデルを作る、というのもあり得るのではないか。
